ThoughtWorks 又发明新概念了。

Harness Engineering 是时下热度很高的概念，但是这里面涉及的、与 agent 系统相关的概念还不算清晰，可以系统看看[这篇文章](https://martinfowler.com/articles/harness-engineering.html)。

### 前馈和反馈

文章借用了控制论里面前馈（Feedfoward）和反馈（Feedback）的概念。

反馈好理解。但是注意反馈要闭环，比如 lint 检查，检查的结果要以**模型友好**的方式反馈给 agent，这里面还是有一些优化。

所谓前馈，其实就是在具体编码需要参考的一系列规则、约束、设计原则等，这个我们也有不陌生。

![harness overview](../images/harness-overview.png)

不过前馈和反馈要同时存在，并且互相作用才能真正发挥作用。

> Separately, you get either an agent that keeps repeating the same mistakes (feedback-only) or an agent that encodes rules but never finds out whether they worked (feed-forward-only).

为什么只有反馈没有前馈，就会导致 agent 反复犯错？

如果只有反馈，agent 虽然在编码-犯错-验证-纠错的循环上逐步前进，但依然可能：

- 只是在局部修补
- 未必抽象出“错误模式”
- 没有把失败总结成下次一开始就该遵守的策略

我想如果给 agent 足够的时间和资源，它最终能找到正确的路径，但效率很低。

所以关键的做法是：

**把反馈转化为前馈**。

### Harnessability

Harnessability 是一个新概念。简单来说，这个概念衡量了一个项目**可以实现 harness 的程度**。

比如：

| Codebase property | Harness capability unlocked |
| --- | --- |
| Strongly typed language | Type-checker as a free, always-on sensor |
| Clear module boundaries | Architectural constraint rules |
| Consistent framework usage | Framework patterns do implicit steering |
| Good naming and modularity | Inferential sensors work better on legible code |
| High test coverage | Behaviour feedback closable via test + mutation testing |

对于新项目和老项目，这些指标显然可满足的难度是不同的，要依据自己项目特点来评估。

细想来，这些也逃不开传统软件工程的范畴。

这也是为什么在 agent 时代，编码本身不那么重要了，但是软件工程的思想和实践会越来越重要。

在没有 agent 之前，工程师还可以**以没有时间编码**为借口，忽略软件工程方面的实践，但是现在这些借口都没有了。

### 人能干什么？

> A good harness should not necessarily aim to fully eliminate human input, but to direct it to where our input is most important.

构建基于前馈和反馈系统的 harness 是很有难度的一件事，有经验的工程师应该着眼于此，让 agent 的效率得到提升。人的经验输入更为重要，而不是相反。

这些概念仔细品来都不是新鲜事物，但是明确的概念和分类，提供了**多个思考维度**，让我们对事物评估和判断更清晰有据。
