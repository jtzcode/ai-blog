今天看到一篇与 agent 记忆管理相关的[文章](https://arxiv.org/pdf/2604.02734)。

传统的记忆模式通常关注：如何压缩和检索。这篇文章从 agent 在长程任务的表现上失败的原因入手，区分**两类模式**，进而区分**两种记忆**。

作者把它们叫作：

**Progress Drift**：从全局看，agent 虽然还在行动，但慢慢偏离主线，开始在无关步骤上打转。
**Feasibility Violation**：从局部看，agent 选了当前状态下**根本不可执行**的动作。

这两个问题在很多现有 agent 实现里被混成一个“统一记忆”来处理，但作者的意见是：这两类问题需要的不是同一种记忆。

因为它们对**推理的要求**不同：

`Progress` 需要的是模糊、语义化、阶段性的指导。
`Feasibility` 需要的是严格、离散、带硬边界的约束验证。（举例：你不能删掉现有 API 字段，因下游依赖还在；或者，你不能提交代码前跳过必需测试）

为此，论文提出的是一个 `Neuro-Symbolic Dual Memory` 框架。它把记忆拆成两部分：

1. Progress Memory

这部分来自**成功轨迹**。系统把成功完成任务的历史轨迹提炼成 `procedural blueprints`，也就是一组阶段化的高层子目标。

再把每个阶段作为 `progress anchor`。执行时，当前 anchor 会驱动系统去检索与“当前阶段”匹配的动作片段，而不是整段历史轨迹。

这里有两个关键词很关键：

- `blueprint`：不是逐步动作，而是阶段性的做事框架
- `anchor`：当前执行到哪个阶段，用它来约束检索范围和行动参考

2. Feasibility Memory

这部分来自失败转移。系统从失败动作中归纳出一组可执行的 `Python verifier functions`，用来在动作真正执行前做硬逻辑检查和 `precondition validation`。

这里不是让 LLM “自己理解一下这个动作合不合理”，而是把失败经验蒸馏成：**可执行验证器**，是一种“硬约束“。

场景的约束可以是：
- `lint / static checks`
- `test templates`
- `contract validators`
- `preflight checks`

总体来看，目前主流的 Coding Agents 里面也有类似的机制。在 Progress 那块，有 CLAUDE.md 和 AGENT.md 去承担；

而 Feasibility 这块，比如 CC 的 hooks 机制可以在特定时机强制执行一些检查，但并没有从记忆的角度，将失败经验自动内化成这些 hooks 的执行。

所谓「知行合一」，真正的记忆，需要配合对应的 action 来体现在“记住“这个效果。说白了，这也还是 harness 的一部分。
