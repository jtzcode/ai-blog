今天来研读另一篇关于 IH 的[文章](https://arxiv.org/abs/2604.09443)。

## 全局地图
### 一句话摘要
这篇论文要证明的不是模型完全不会遵守指令，而是模型并没有稳定掌握“多来源约束发生冲突时，应该依据显式权限关系来裁决”的能力。传统 instruction hierarchy 把权限绑在固定角色上，ManyIH 则试图把权限拆到 instruction 级别，并且在推理时动态注入。

### 段落分类
- [骨] 摘要与引言：提出固定层级瓶颈
- [骨] 第 2-3 节：定义 instruction / privilege / tiers，并提出 PPI
- [骨] 第 4-6 节：benchmark 设计与实验结论
- [肌] 第 5 节：benchmark 构建流水线
- [筋] 第 7-8 节：相关工作、结论与伦理

## 逐段伴读记录
### 第 1 段：旧 IH 的瓶颈
#### 翻译
作者认为，传统 IH 主要依赖固定 role label，如 system > developer > user > tool。但真实 agent 会同时接收 system prompt、skill、memory、tool output、retrieved evidence、other agents 等多种来源，角色粒度已经不足以表达真实权限结构。
#### 结构标注
这是问题定义段。它把“agent 世界更复杂”压缩成固定且少数层级的瓶颈。
#### 注疏
关键不是 hierarchy 这个词，而是“权限从 role label 中拆出来”。旧范式把权限嵌在角色模板里；新范式把权限当作 instruction-level metadata。
#### 碰撞记录
读者抓到的关键点是：归属同一角色的来源依然可能冲突，因此角色不是足够细的权限单位。

### 第 2 段：ManyIH 怎么做动态权限
#### 翻译
ManyIH 用 Privilege Prompt Interface 在 prompt 中显式标注每条 instruction 的 privilege。Ordinal 形式使用 [[Privilege N]]...[[/Privilege]]，数字越小权限越高；scalar 形式使用 [[z=N]]...[[/z]]，数字越大权限越高。
#### 结构标注
这是定义性创新段。论文要测的不是模型会不会背 role template，而是会不会在运行时读懂 instruction-level privilege。
#### 注疏
绝对数值本身没有语义，只有相对顺序有意义。理论上 91 > 55 > 12 与 9 > 5 > 1 应当等价；实验却表明模型对表面形式和具体数字都敏感，这说明模型尚未学会稳定的“排序不变性”。
#### 碰撞记录
读者确认：这里的 privilege 已经泛化成约束冲突中的裁决权，而不止是安全语境下的访问控制。

### 第 3 段：PPI 为什么用 tag-based prompt，而不是系统外硬控制
#### 翻译
论文之所以把 privilege 做成 prompt interface，而不是外部规则引擎直接替模型裁决，是因为作者要测的是“模型内的冲突解决能力”，而不是“系统外已经替你裁完以后模型会不会照抄”。
#### 结构标注
这是方法论分界段。ManyIH 研究的是 model-side resolution，不是 orchestration-side hard control。
#### 注疏
如果外部系统先把所有冲突都归并好，再只把 winning instructions 发给模型，那么被测的就不再是 privilege reasoning，而只是普通 instruction following。ManyIH 让模型直接面对冲突集合，观察它能否：识别冲突、比较 privilege、保留 winners、同时满足非冲突约束。
#### 碰撞记录
结论：tag-based PPI 不是最可靠的生产控制面，但它是最直接的实验接口，用来暴露当前模型在权限裁决上的真实短板。

### 第 4 段：Benchmark 怎么构出来
#### 翻译
ManyIH-Bench 有两个子集。Coding subset 把 MBPP 题目与冲突 style instructions 结合，要求代码既功能正确，又遵守 winning style constraints。Instruction-following subset 则基于 46 个真实 agent 场景，引入 privilege-annotated conflicting constraints。
#### 结构标注
这是证据基础段，目的是保证“难度主要来自权限裁决”，而不是来自任务本身太杂乱。
#### 注疏
作者专门做了几件事来隔离变量：
- 非对抗式 prompt，避免把攻击鲁棒性混进来
- 约束级独立验证，而不是只看整体印象
- coding 子集固定 winning instructions 数量，只增加 tiers 与 conflicts
- IF 子集用真实 agent prompt，而不是纯合成 toy tasks
#### 碰撞记录
最值得注意的是 human validation：100 个随机约束检查中，81 个 faithful，11 个 unclear，8 个 incorrect。也就是说 benchmark 不是完美的，但作者至少正面承认并量化了噪声。

### 第 5 段：实验结果说明了什么
#### 翻译
最佳模型仅约 42.7% overall accuracy。GPT-5 系统卡在两层 IH 上超过 99%，但在 ManyIH-Bench 上只有 39.5%。随着 tiers 增加，几乎所有模型都稳定下降。换个 privilege 表示法，GPT-5.4 和 Opus 4.6 都会掉约 8%。
#### 结构标注
这是全文最硬的证据段：many-tier privilege reasoning 是一个独立且尚未解决的能力问题。
#### 注疏
coding subset 的拆分很有力：功能正确率普遍高，但 style accuracy 低。这说明模型不是不会解题，而是在冲突约束里不会稳定挑出该听谁。
#### 碰撞记录
读者将其理解为：今天很多 frontier model 更像“能把活干完的执行者”，而不是“能稳定判断规则冲突中谁说了算的制度性 agent”。

### 第 6 段：更细的分析
#### 翻译
更长的 chain-of-thought 不自动带来更高准确率；提高 reasoning effort 有帮助，但不能从根本上解决 ManyIH 问题。scalar privilege 值的小扰动即使不改变排序，也会引起样本翻转。
#### 结构标注
这是机制诊断段。论文说明问题不只是算力或思考长度不够，而是 privilege representation 没有被模型稳健抽象。
#### 注疏
如果模型真正掌握了排序规则，那么同构表示之间不应有显著差异，保持顺序的数值微调也不应影响结果。现实恰恰相反，因此 ManyIH 暴露的是 representation + reasoning 的双重脆弱。
#### 碰撞记录
读者接受这一点：模型并不是完全不认识这些 tag，而是在“临时被 prompt 教会”与“真正内化抽象规则”之间，停留在一个很脆的中间状态。

## 全文复盘
### 理解轨迹
- 通过：论文问题定义很清楚，fixed/few-tier bottleneck 站得住。
- 停留：privilege assignment 的正当性不在论文范围内，作者把它假设为外部已给。
- 抵抗：tag-based PPI 作为生产机制并不可靠，但作为实验接口恰恰足够有力，因为它能暴露模型内部裁决能力的不足。

### 终局问题
如果未来 agent 的真实控制面必须同时依赖外部治理模块与模型内裁决能力，那么权限冲突的“立法权”究竟应该掌握在谁手里，才不会把错误的秩序稳定执行下去？

### 术语表
- Instruction Hierarchy / 指令层级：冲突指令之间按权限裁决的规则
- Privilege / 权限值：约束冲突中的裁决优先级，不限于安全语境
- ManyIH / 多层级指令层级：支持任意数量权限层级的 instruction conflict resolution 范式
- PPI / Privilege Prompt Interface：把权限值编码进 prompt 的表示接口
- Ordinal interface / 序数接口：数字越小权限越高
- Scalar interface / 标量接口：数字越大权限越高
- Active constraints / 生效约束：冲突组中最终胜出的 instructions
- Suppressed constraints / 被抑制约束：因权限较低而失效的 instructions
- Fixed/few-tier bottleneck / 固定少层级瓶颈：传统 role-based IH 只能处理很少的预定义权限层次
