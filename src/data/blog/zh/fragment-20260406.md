强大的 Coding Agent 已被广泛使用，Vibe Coding 的项目层出不穷，写出原型乃至产品初始形态，agent 似乎已经可以胜任。

但是对于**长期迭代**的项目呢？agent 的能力演化到今天的地步，时间并没有很长，因此来自长期迭代的项目的考验还不是很充分。[这篇文章](https://arxiv.org/abs/2603.24755)对此进行了研究。

针对 agent 的评估，不应该只看是否一次能实现需要的功能，还需要考察项目的代码的长期质量（如会不会出现代码冗余、复杂度显著提升等现象），这需要配套的 benchmark。

初始的 prompt，比如详细的设计 spec，可以显著提高早期版本的质量，但依然不能避免长期迭代后，代码质量的退化。agent 需要关注一个项目代码质量的**持续治理**。

真实的场景是，agent 每一轮都**继承**自己的旧代码，而旧代码又影响下一轮修改方式；于是局部补丁越来越倾向于继续补在已有的、复杂度较高的地方，最终形成「结构坍缩」。

这一点上，其实 agent 并不如一个代码素养良好的团队。

那篇文章用了两个指标衡量代码质量：

> 结构侵蚀（Structural Erosion）：代码库的功能是否越来越集中到少数高复杂度函数里。

以及

> 冗余度（Verbosity）：新增代码里有多少是重复、可压缩、信息密度低的“多余”代码。

这让我想到《软件设计的哲学》一书对软件复杂性的讨论。对于 agent 来说，基本不存在「认知负载」以及「未知的未知」的因素。上述两个指标可以归为这两类。

至于「变更放大」，则是实打实的存在。这是不是说明，这类复杂性的引入会可能**显著提高 token 使用量**呢？这个就是「软件复杂性经济学」了。

因此，在 harness engineering 的实践中，不只检验“能不能过”，还要检验“是不是在变坏”。

比如，除了基本的单元测试和lint检查，还可以：

- 重复代码比例检查
- 单函数复杂度阈值检查
- 文件膨胀速率检查
- **先评估是否需要小范围重构，再添加新功能**
- ...

agent 也需要「战略性编程」，即目标要围绕**始终产出高质量、低复杂性的软件系统**，而不是实现功能。
